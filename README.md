# Netcode Lab — 딜레이 vs 롤백 넷코드 실험실

1D 패링/타격 시뮬레이터로 딜레이 넷코드와 롤백 넷코드의 동작 차이를 직접 관찰하는 도구입니다.

## 실행 방법

### 1. 가상 피어 시뮬레이션 모드 (기본, 서버 불필요)

`public/` 폴더는 ES 모듈(`import`)을 사용하므로 `file://`로 직접 열면 브라우저가 막습니다.
아무 정적 서버로 열어주세요.

```bash
cd netcode-lab/public
npx serve .
# 또는: python3 -m http.server 5173
```

브라우저에서 열리는 주소로 접속하면 "가상 피어 시뮬레이션" 모드가 기본으로 켜져 있습니다.
같은 키보드로 P1(A/D 이동, F 공격)과 P2(←/→ 이동, / 공격)를 동시에 조작하면서,
두 캔버스(P1 관점 / P2 관점)를 나란히 비교해 보세요.

### 2. 실제 WebSocket 모드 (두 클라이언트로 진짜 테스트)

```bash
cd netcode-lab/server
npm install
npm start   # ws://localhost:8080 에서 릴레이 서버 실행
```

그다음 브라우저 탭을 **두 개** 열어 각각 위 정적 서버 주소로 접속하고,
연결 방식을 "실제 WebSocket"으로 바꾼 뒤 "연결"을 누르세요. 먼저 연결한 탭이 P1, 나중에
연결한 탭이 P2가 됩니다. 두 클라이언트가 모두 접속하면 서버가 시작 신호를 보내 양쪽이
frame 0부터 동시에 시작하며, 상대가 나가면 재접속할 때까지 대기했다가 처음부터 다시
시작합니다. 각 탭은 A/D 이동, F 공격으로 자신의 캐릭터만 조작합니다.

> 이 저장소에는 `npm install`을 실행하지 않았습니다 (이 환경은 외부 네트워크에 접근할 수
> 없습니다). 로컬에서 `npm install`을 한 번 실행해야 `ws` 패키지가 설치됩니다.

## 무엇을 실험할 수 있나요

- **지연(Latency) 슬라이더**: 딜레이 모드에서 올릴수록 입력이 무거워집니다(내부적으로 핑의
  절반만큼 입력 지연 프레임이 자동 계산됩니다). 롤백 모드에서는 조작감 자체는 그대로지만,
  상대 캐릭터가 정정될 때 위치가 튀는(순간이동) 정도가 커집니다.
- **지터(Jitter) 슬라이더**: 딜레이 모드에서 큰 지터를 주면 화면 하단에 "정지" 표시와 함께
  게임이 버벅입니다(원격 입력이 예상보다 늦게 도착). 롤백 모드에서는 하단에 "롤백: N프레임
  재시뮬레이션"이 더 자주, 더 큰 폭으로 뜹니다.
- **패킷 손실(Loss) 슬라이더**: 매 패킷마다 최근 8프레임 치 입력을 중복 전송하므로, 어느 정도
  손실률까지는 다음 패킷이 빈 곳을 메꿔 자연 복구됩니다. 손실률을 아주 높이면(40%+) 그 복구
  한계를 넘어서면서 정지/보정이 눈에 띄게 잦아지는 것도 관찰할 수 있습니다.

## 구조

```
public/src/
  gameState.js          정수 기반 결정론적 시뮬레이션 (step 함수)
  inputBuffer.js         프레임 링 버퍼
  networkImpairment.js   지연/지터/손실을 주입하는 가상 네트워크 레이어
  transports/
    loopbackTransport.js 한 탭 안에서 두 피어를 시뮬레이션 (가상 네트워크 통과)
    websocketTransport.js 실제 WebSocket 전송 (그 위에 가상 네트워크도 추가 적용 가능)
  netcode/
    delayRunner.js       딜레이 넷코드
    rollbackRunner.js    롤백 넷코드 (예측 + 롤백 + 재시뮬레이션)
  renderer.js            캔버스 렌더링
  app.js                 UI 배선, 입력 폴링, 고정 타임스텝 루프
server/
  server.js              2인용 WebSocket 릴레이 서버
```

두 넷코드 러너 모두 `transport` 인터페이스(`send`/`onReceive`/`setParams`)에만 의존하므로,
루프백이든 실제 WebSocket이든 동일한 넷코드 로직이 그대로 재사용됩니다.

## 알려진 한계

- 히스토리 링 버퍼가 90프레임(약 1.5초)이라, 지연을 그 이상으로 올리면 롤백이 스냅샷을
  찾지 못해 보정을 포기하고 그냥 진행합니다 (`inputBuffer.js`의 `HISTORY_SIZE` 조정 가능).
- 패킷 손실률이 극단적으로 높으면(약 60%+) 8프레임 중복 전송으로도 완전히 복구되지 않을 수
  있습니다 — 이 역시 실제 UDP 기반 네트코드가 겪는 현실적인 한계와 같습니다.
